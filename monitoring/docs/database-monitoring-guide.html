Database Monitoring Guide

OTel Collector Receivers vs Traditional Exporters

1 Enable MySQL Instrumentation

Mount a custom my.cnf into MySQL to enable slow query logging and performance schema.

⚙
monitoring/database/my.cnf
[mysqld]
slow_query_log              = 1                        # Enable slow query logging
slow_query_log_file         = /var/log/mysql/slow.log  # Output path
long_query_time             = 0.5                      # Threshold: > 500ms
log_queries_not_using_indexes = 1                      # Log full-scan queries
performance_schema          = ON                       # Enable stats tables

2 OTel Collector — mysql receiver

The mysql receiver connects via SQL, scrapes SHOW GLOBAL STATUS and performance_schema tables every 15 seconds.

OTel Receiver mysql receiver config▼
receivers:
  mysql:
    endpoint: 127.0.0.1:3306
    username: ${env:DB_USERNAME}
    password: ${env:DB_PASSWORD}
    collection_interval: 15s
    statement_events:
      digest_text_limit: 120
      time_limit: 24h
      limit: 250
    metrics:
      mysql.commands:             { enabled: true }
      mysql.query.count:           { enabled: true }
      mysql.query.slow.count:      { enabled: true }
      mysql.connection.count:      { enabled: true }
      mysql.connection.errors:     { enabled: true }
      mysql.statement_event.count: { enabled: true }
      mysql.statement_event.wait.time: { enabled: true }

Key metrics: mysql_commands_total, mysql_query_count_total, mysql_threads, mysql_table_io_wait_count_total, mysql_statement_event_count, mysql_buffer_pool_pages

logs filelog receiver — tailing slow.log▼
receivers:
  filelog/slow_query:
    include: [/var/log/mysql/slow.log]
    multiline:
      line_start_pattern: '^# Time:'    # group multi-line entries
processors:
  resource/mysql:
    attributes:
      - { key: service.name, value: mysql, action: upsert }

Requires shared volume (emptyDir) between MySQL container and OTel sidecar. OTel sidecar runs as runAsUser: 999 (mysql user) to read the log file.

1 Enable PostgreSQL Instrumentation

PostgreSQL needs pg_stat_statements extension and optionally slow query logging via postgresql.conf.

⚙
postgresql.conf
# Load pg_stat_statements extension for query-level stats
shared_preload_libraries = 'pg_stat_statements'

# Slow query logging
log_min_duration_statement = 500       # Log queries > 500ms
log_statement              = 'none'    # Don't log all statements
log_destination            = 'csvlog'  # Structured CSV format
logging_collector          = on
log_directory              = '/var/log/postgresql'
log_filename               = 'slow.log'
log_line_prefix            = '%t [%p]: db=%d,user=%u '

Then run CREATE EXTENSION IF NOT EXISTS pg_stat_statements; in each database.

1 Enable SQL Server Instrumentation

SQL Server has built-in Dynamic Management Views (DMVs) — no config file needed. Just ensure the monitoring user has VIEW SERVER STATE permission.

⚙
SQL Server Setup
-- Grant monitoring permissions
CREATE LOGIN otel_monitor WITH PASSWORD = 'SecurePass!';
GRANT VIEW SERVER STATE TO otel_monitor;
GRANT VIEW ANY DEFINITION TO otel_monitor;

-- Optional: enable Query Store (built-in slow query tracking)
ALTER DATABASE mydb SET QUERY_STORE = ON;
ALTER DATABASE mydb SET QUERY_STORE (
  OPERATION_MODE = READ_WRITE,
  MAX_STORAGE_SIZE_MB = 256,
  INTERVAL_LENGTH_MINUTES = 5
);

SQL Server uses DMVs instead of performance_schema: sys.dm_os_performance_counters, sys.dm_exec_query_stats, sys.dm_os_wait_stats

Key metrics: sqlserver_batch_request_rate, sqlserver_user_connection_count, sqlserver_page_buffer_cache_hit_ratio, sqlserver_lock_wait_rate, sqlserver_processes_blocked, sqlserver_database_io_read_latency

Internally queries sys.dm_os_performance_counters and sys.dm_os_wait_stats

logs SQL Server Error Log / slow queries▼
# SQL Server doesn't have a native slow query log file like MySQL.
# Options for query-level logging:

# Option A: Extended Events (XEvents) → file → filelog receiver
receivers:
  filelog/sqlserver:
    include: [/var/opt/mssql/log/errorlog]    # Error log
    multiline:
      line_start_pattern: '^\\d{4}-\\d{2}-\\d{2}'

# Option B: Query Store → custom SQL receiver (advanced)
# Use sqlquery receiver to periodically query sys.query_store_*

SQL Server has no built-in slow query log file. Use Extended Events (XEvents) to capture slow queries to a file target, then tail with filelog receiver. Or use Query Store (built-in since SQL Server 2016) and query it directly.

⚙
Custom Query Example
# sql_exporter collector config
collector_name: mssql_performance
metrics:
  - metric_name: mssql_batch_requests
    type: counter
    help: "Batch requests/sec"
    values: [cntr_value]
    query: |
      SELECT cntr_value
      FROM sys.dm_os_performance_counters
      WHERE counter_name = 'Batch Requests/sec'

1 Enable Oracle Instrumentation

Oracle has built-in instrumentation via V$ views and ASH (Active Session History). Grant monitoring permissions to the OTel user.

⚙
Oracle Setup
-- Create monitoring user with required permissions
CREATE USER otel_monitor IDENTIFIED BY "SecurePass";
GRANT CREATE SESSION TO otel_monitor;
GRANT SELECT_CATALOG_ROLE TO otel_monitor;
GRANT SELECT ON V_$SESSION TO otel_monitor;
GRANT SELECT ON V_$SYSSTAT TO otel_monitor;
GRANT SELECT ON V_$SQL TO otel_monitor;
GRANT SELECT ON DBA_TABLESPACE_USAGE_METRICS TO otel_monitor;

-- Optional: enable AWR snapshots for historical analysis
EXEC DBMS_WORKLOAD_REPOSITORY.MODIFY_SNAPSHOT_SETTINGS(
  interval => 15,        -- minutes
  retention => 43200    -- 30 days
);

Oracle exposes stats via V$ views: V$SYSSTAT, V$SESSION, V$SQL, V$SYSTEM_WAIT_CLASS, DBA_TABLESPACE_USAGE_METRICS

Two Layers of Database Metrics

This is the key insight: there are two completely different layers of database metrics, and only one has shared semantic conventions.

✓
Layer 1: Shared Semantic Conventions

The OTel SDK in your application code auto-instruments database client calls. These follow the db.client.* semantic conventions.

Stable Metric (same for ALL databases)

# This metric is identical whether you use MySQL, PostgreSQL, MSSQL, or Oracle
db.client.operation.duration  # Histogram (seconds)

# The database type is just an attribute:
  db.system.name = "mysql"          # or "postgresql", "mssql", "oracle"
  db.operation.name = "SELECT"      # the SQL operation
  db.namespace = "store"           # database/schema name
  db.collection.name = "orders"    # table name
  db.response.status_code = "42P01" # DB-specific error code

Connection Pool Metrics (experimental)

db.client.connection.count            # UpDownCounter
db.client.connection.idle.max         # UpDownCounter
db.client.connection.max              # UpDownCounter
db.client.connection.pending_requests # UpDownCounter
db.client.connection.timeouts         # Counter
db.client.connection.create_time      # Histogram
db.client.connection.wait_time        # Histogram
db.client.connection.use_time         # Histogram
db.client.response.returned_rows      # Histogram

Key benefit: Switch from PostgreSQL to MySQL? Your dashboards still work — same metric name, same attributes. Just filter by db.system.name.

❌
Layer 2: No Shared Convention

The OTel Collector receivers scrape the database engine directly. Each database has fundamentally different internals, so metric names are completely different.

Why no convention? Because these come from completely different sources: MySQL SHOW GLOBAL STATUS / performance_schema, PostgreSQL pg_stat_* views, SQL Server sys.dm_* DMVs, Oracle V$ views. The underlying concepts don't map 1:1.

What this means in practice

If you want a "universal database dashboard":

  • Use Layer 1 (db.client.operation.duration)
  • One dashboard works for ALL databases
  • Shows: query latency, throughput, errors
  • Filter by db.system.name attribute
  • Limitation: only sees what the app sees (client-side)

If you want deep DB engine internals:

  • Use Layer 2 (database-specific receivers)
  • Each database needs its own dashboard
  • Shows: buffer pools, locks, I/O waits, replication, tablespaces
  • Metric names are unique per database engine
  • Can't build a "one dashboard fits all"

Most production setups use both layers: Layer 1 for application-level SLIs (latency, error rate) and Layer 2 for DBA-level deep diagnostics (buffer pool hit ratio, lock contention, slow query analysis).

When to use which approach?

OTel Collector is best when:

  • Already using OTel for application tracing/metrics
  • Need unified metrics + logs pipeline
  • Cross-cluster push model (no Prometheus in app cluster)
  • Want a single standard across all databases

Prometheus Exporter is best when:

  • Pure Prometheus stack, everything in one cluster
  • Want pre-built community dashboards immediately
  • Need custom SQL queries for app-specific metrics
  • Using Prometheus Operator with ServiceMonitor CRDs
