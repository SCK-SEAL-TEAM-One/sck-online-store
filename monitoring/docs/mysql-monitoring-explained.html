MySQL Monitoring Explained

1 Enable MySQL Instrumentation

Everything starts with monitoring/database/my.cnf, mounted into MySQL as /etc/mysql/conf.d/slow-query.cnf.

⚙
my.cnf Configuration 
# monitoring/database/my.cnf
[mysqld]
slow_query_log              = 1                        # Enable slow query logging
slow_query_log_file         = /var/log/mysql/slow.log  # Output file path
long_query_time             = 0.5                      # Threshold: queries > 500ms
log_queries_not_using_indexes = 1                      # Also log full-scan queries
performance_schema          = ON                       # Enable internal stats tables

This unlocks two data sources from MySQL:

  • Slow query log file — raw text log of every slow / unindexed query
  • Performance Schema tables — structured stats like events_statements_summary_by_digest, table_io_waits_summary_by_table

2 Architecture — OTel Collector Approach

A single OTel Collector sidecar handles metrics and logs in one unified pipeline.

3 OTel Collector Configuration

The sidecar config has two receivers feeding two pipelines through one exporter.

metrics mysql receiver — SQL-based scraping ▼ 
receivers:
  mysql:
    endpoint: 127.0.0.1:3306          # localhost in same pod
    username: ${env:DB_USERNAME}       # from K8s secret
    password: ${env:DB_PASSWORD}
    collection_interval: 15s
    statement_events:               # queries performance_schema
      digest_text_limit: 120
      time_limit: 24h
      limit: 250

Internally runs SQL like SHOW GLOBAL STATUS, SELECT ... FROM performance_schema.events_statements_summary_by_digest, etc.

Produces metrics: mysql_commands_total, mysql_query_count_total, mysql_threads, mysql_table_io_wait_count_total, mysql_statement_event_count, and 28 more.

logs filelog receiver — tailing slow.log ▼ 
receivers:
  filelog/slow_query:
    include:
      - /var/log/mysql/slow.log    # shared emptyDir volume
    multiline:
      line_start_pattern: '^# Time:' # each entry starts with timestamp

Groups multi-line slow query entries into single log records:

# Time: 2026-03-17T22:43:24.799660Z
# User@Host: user[user] @ [10.0.1.209]  Id: 10
# Query_time: 0.52  Lock_time: 0.00  Rows_sent: 150  Rows_examined: 15000
SELECT * FROM orders WHERE status = 'pending';
Pipelines — how data flows through ▼ 
service:
  pipelines:
    metrics:                         # Pipeline 1: MySQL stats → Prometheus
      receivers:  [mysql]
      processors: [resource/mysql]
      exporters:  [otlphttp]          # → monitoring cluster NLB :4318
    logs:                             # Pipeline 2: slow.log → Loki
      receivers:  [filelog/slow_query]
      processors: [resource/mysql]    # adds service.name=mysql
      exporters:  [otlphttp]          # → same endpoint, routed to Loki

5 Alternative: mysqld_exporter

The Prometheus community mysqld_exporter is the traditional approach. Key difference: it's pull based vs OTel's push model.

⚙
K8s Deployment (sidecar)
- name: mysqld-exporter
  image: prom/mysqld-exporter:latest
  env:
    - name: DATA_SOURCE_NAME
      value: "user:password@(127.0.0.1:3306)/"
  args:
    - --collect.info_schema.tables
    - --collect.perf_schema.tableiowaits
    - --collect.perf_schema.eventsstatements
  ports:
    - containerPort: 9104
⚙
Prometheus Configuration
scrape_configs:
  - job_name: mysql
    scrape_interval: 15s
    static_configs:
      - targets: ['mysql:9104']    # K8s service pointing to exporter

Prometheus must be able to reach the exporter. In a cross-cluster setup, this requires either:

  • Prometheus in the same cluster as MySQL
  • Prometheus federation or remote-write from a local Prometheus
  • ServiceMonitor CRD if using Prometheus Operator

6 Head-to-Head Comparison

OTel Collector mysql receiver mysqld_exporter
Model push OTLP → remote write pull Prometheus scrapes /metrics
Metrics mysql_commands_total, mysql_threads, etc. mysql_global_status_com_select, mysql_global_status_threads_connected
Logs Yes — filelog receiver tails slow.log No — metrics only
Cross-cluster Easy — push to NLB endpoint Hard — need Prometheus in app cluster or federation
Community dashboards Custom (must match OTel metric names) Grafana ID 7362, 14057 (pre-built, battle-tested)
Config complexity YAML receivers/pipelines/exporters One env var + collector flags
Ecosystem fit Best when already using OTel (unified pipeline) Best for pure Prometheus stack
Maintained by OpenTelemetry (CNCF) Prometheus (CNCF)

Why OTel Collector for this project?

  • Unified pipeline — one sidecar handles both metrics AND slow query logs
  • Cross-cluster push — workshop cluster pushes to monitoring cluster NLB, no need for Prometheus in app cluster
  • Consistent with other services — store-service, point-service, thirdparty all use OTLP already
  • Single exporter — everything goes through one OTLP/HTTP endpoint

If you were running a pure Prometheus stack with everything in one cluster, mysqld_exporter + a separate log shipper (Promtail/Alloy) would be a perfectly valid choice with better community dashboard support.
